### Tap-Tap Interaction (User Decisions D-01 through D-04)

1. **First tap (select source square):**
   - User taps square with piece of correct color
   - Square receives blue 4px border (selected highlight)
   - No visual indication of valid destination squares (trust-based gameplay D-04)

2. **Tap same square (deselect):**
   - User taps selected square again
   - Blue highlight removed
   - Selection cleared

3. **Second tap (move piece):**
   - User taps destination square
   - chess.js validates move (internal only, no UI indicator)
   - If valid: move executes, FEN updates, move added to history, board re-renders
   - If invalid: selection cleared silently (no error toast in trust mode)

**Pre-populated from:** CONTEXT.md (D-01, D-02, D-03, D-04)

### Auto-Scroll Move History (User Decision D-16)

- After each move, latest move in history auto-scrolls into view
- User can manually scroll up to review earlier moves
- Auto-scroll uses `scrollIntoView({ behavior: 'smooth', block: 'end' })`
- Triggered on moves array length change

**Pre-populated from:** CONTEXT.md (D-16, D-17), RESEARCH.md (Pattern 4)

## State & Persistence

### Game State Structure

Stored as single SQLite row in `chess_games` table (id = 1):

```sql
CREATE TABLE IF NOT EXISTS chess_games (
  id INTEGER PRIMARY KEY CHECK (id = 1),
  fen TEXT NOT NULL DEFAULT 'rnbqkbnr/pppppppp/8/8/8/8/PPPPPPPP/RNBQKBNR w KQkq - 0 1',
  moves TEXT NOT NULL DEFAULT '[]',
  white_player_id INTEGER,
  black_player_id INTEGER,
  created_at INTEGER DEFAULT (unixepoch()),
  updated_at INTEGER DEFAULT (unixepoch()),
  FOREIGN KEY (white_player_id) REFERENCES family_members(id) ON DELETE SET NULL,
  FOREIGN KEY (black_player_id) REFERENCES family_members(id) ON DELETE SET NULL
);
```

**FEN notation:** Standard Forsyth-Edwards Notation for board position (pieces, castling rights, en passant, halfmove clock, fullmove number)

**Move history:** JSON array of Standard Algebraic Notation (SAN) strings: `["e4", "e5", "Nf3", "Nc6"]`

**Persistence:** Load on mount, save after each move, reset on new game, undo removes last move and reverts FEN

**Pre-populated from:** CONTEXT.md (D-18, D-19, D-20, D-21), RESEARCH.md (Pattern 3)

## Implementation Notes

### Key Decisions Locked In

| Decision ID | Rule | Rationale |
|-------------|------|-----------|
| D-01 | Tap-tap piece movement | Simpler for touchscreen than drag-and-drop; no precision issues |
| D-02 | Blue border on selected square | Visual feedback for first-tap confirmation |
| D-03 | Tap same square to deselect | Clear cancellation pattern |
| D-04 | No valid move highlighting | Trust-based gameplay; users learn rules through play |
| D-05 | Unicode chess symbols | Zero asset management; crisp rendering; 10KB+ smaller than SVG |
| D-06 | Neutral gray board squares | Modern aesthetic; high contrast with pieces |
| D-07 | Unicode white/black distinction | No custom CSS coloring needed |
| D-08 | 8×8 grid with optional coordinates | Standard chess layout |
| D-09 | Sidebar layout (right of board) | Vertical flow fits touchscreen better than horizontal |
| D-10 | Minimal control set | Focus on core gameplay; reduces clutter |
| D-11 | Tap player badge for selection | Tappable badges reduce modal launch friction |
| D-12 | Confirmation dialog for new game | Prevents accidental game loss |
| D-13 | Board square aspect ratio, sidebar fills remainder | Maintains board integrity on responsive resize |
| D-14 | Move history in sidebar below controls | All UI in one column |
| D-15 | Vertical move list format | Easier to read on narrow sidebars |
| D-16 | Always auto-scroll to latest move | Keeps ongoing games in view |
| D-17 | Manual scroll allowed | Users can review past moves |
| D-18 | SQLite with FEN + moves JSON | Atomic state, simpler persistence |
| D-19 | Single game at a time | v1 scope; simplifies state management |
| D-20 | Load on mount, save after each move | Persistence across browser restart (CHESS-12) |
| D-21 | Undo removes from DB and reverts FEN | Full game state consistency |
