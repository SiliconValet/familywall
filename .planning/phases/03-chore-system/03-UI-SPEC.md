## Spacing Scale

Declared values (Tailwind default scale, all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| 1 | 4px | Icon gaps, minimal inline padding |
| 2 | 8px | Compact element spacing, internal button padding, card gaps |
| 4 | 16px | Default element spacing, form field gaps, card internal padding |
| 6 | 24px | Section padding, major element separation |
| 8 | 32px | Layout gaps between major sections |
| 12 | 48px | Major section breaks, page-level spacing |
| 16 | 64px | Large page-level spacing (if needed) |

**Exceptions for this phase:**
- Touch targets: Minimum 48px (12 units) for all interactive elements
  - Chore completion checkbox: `min-h-12 min-w-12` (48px) per CHOR-06
  - Add Chore button: `min-h-12` (48px)
  - View toggle buttons (Daily/Weekly): `min-h-12` (48px)
  - Family member picker buttons: `min-h-12` (48px)
  - Dialog close buttons: `min-h-12` (48px)

**Rationale:** CHOR-06 requires 48px minimum for chore completion checkboxes for high-precision interaction. All interactive elements standardized to 48px for consistent touch targets and grid alignment (48px = 12 × 4px base unit).

| **Secondary (30%)** | `--card: oklch(1 0 0)` (white), `--muted: oklch(0.97 0 0)` (light gray) | Chore card backgrounds, completed section, empty states |
| **Accent (10%)** | `--primary: oklch(0.52 0.105 223.128)` (blue) | Reserved for: "Add Chore" button only, focus rings |

- Active chore checkbox: `--border` (neutral)

- Auto-completed chore (missed): `--muted-foreground` with strikethrough

**Border/Input:**
- Border: `oklch(0.922 0 0)` (light gray)
- Input border: `oklch(0.922 0 0)` (matches border)

## Interaction Patterns

### Touch Feedback
All interactive elements provide immediate visual feedback (inherited from Phase 2):

```css
/* Applied to all buttons and tappable elements */
.touch-target:active {
  transform: scale(0.96);
  transition: transform 150ms ease-out;
}
```

**Requirements (FAM-08 + CHOR-12):**
- Feedback appears within 100ms of touch
- Visual change is perceivable (scale + opacity or background color shift)
- No hover states (kiosk is touch-only)

### Chore Completion Flow

**Default (normal tap):**
1. User taps 48px checkbox on chore card
2. Checkbox animates (scale 0 → 1.2 → 1.0, 300ms ease-out) per user-approved decision
3. Chore card fades out over 200ms
4. Chore moves to "Completed" section (expands if previously collapsed)
5. Toast notification appears: "Chore completed" with 5-second "Undo" button
6. `completed_by` defaults to assignee (D-06)

**Override (long-press for "someone else did it"):**
1. User presses and holds checkbox for 1000ms (1 second, per D-17)
2. Haptic feedback (if supported by device)
3. "Who completed this?" family member picker modal appears
4. User taps family member from list
5. Same completion animation as default, but `completed_by` set to selected member

**Undo flow:**
1. User taps "Undo" button in toast within 5 seconds (per D-19)
2. Toast disappears immediately
3. Chore card re-appears in active list with fade-in
4. Chore is removed from "Completed" section
5. Toast notification: "Completion undone"
6. Database status reverted to 'active'

**Design notes:**
- Long-press threshold: 1000ms (D-17)
- Must cancel long-press if finger moves > 10px (scroll gesture)
- Undo toast must have large tap target (48px "Undo" button)
- Position toast near completion action for quick access

### Daily vs Weekly View Toggle

1. User taps "Today" or "This Week" toggle button
2. Active button changes to accent color
3. List filters immediately (no loading state - data already loaded)
4. Empty state adapts to view: "All Done for Today!" vs "No Chores This Week"
5. Completed section respects view filter (only show completed from active view period)

**Design notes:**
- Toggle buttons: Segmented control style, 48px height each
- Active state: Primary accent background, white text
- Inactive state: Muted background, muted foreground text

### Add/Edit Chore Form Flow

**Add Chore (PIN-gated, per D-10):**
1. User taps "Add Chore" button (top of page)
2. PIN modal appears (reuses Phase 2 PinModal component)
3. User enters 4-digit PIN
4. On success: Add Chore modal opens
5. User fills: Title (required), Assignee (required dropdown), Points (optional number), Description (optional textarea), Recurring toggle (optional)
6. If Recurring enabled: Frequency picker appears (Daily/Weekly/Custom Days/Every N Days)
7. User taps "Add Chore" button
8. Form validates: Title and Assignee required
9. On success: Modal closes, chore appears in list, toast "Chore added successfully"
10. On error: Inline validation errors appear, form remains open

**Edit Chore (PIN-gated):**
1. User taps edit icon on chore card (48px touch target with `aria-label="Edit chore"`)
2. PIN modal appears
3. On success: Edit Chore modal opens with pre-filled data
4. User modifies fields
5. User taps "Save Changes" button
6. Same validation as Add flow
7. On success: Modal closes, chore updates in list, toast "Chore updated successfully"

**Delete Chore (PIN-gated):**
1. User taps delete icon on chore card (48px touch target with `aria-label="Delete chore"`)
2. PIN modal appears
3. On success: Delete confirmation dialog opens
4. Dialog shows: "Delete This Chore?" with chore title
5. User taps "Delete" or "Keep Chore"
6. On Delete: Chore removed from list, toast "Chore deleted"

**Design notes:**
- Edit/Delete icons must include ARIA labels for accessibility (`aria-label="Edit chore"` and `aria-label="Delete chore"`)
- Recurring frequency UI only appears when "Recurring" toggle is enabled
- Custom Days shows checkboxes for Mon/Tue/Wed/Thu/Fri/Sat/Sun
- Every N Days shows number input (1-365 range)
- Form modal is vertically scrollable if content exceeds viewport height

### Recurring Chore Generation

**Automatic (server-side, midnight):**
- Node.js cron job runs at 12:01am daily (D-22)
- Generates new instances for recurring chores matching frequency pattern
- Auto-completes yesterday's instance if incomplete: status='auto_completed', completed_by=NULL (D-23)
- No user-visible UI for generation (happens in background)

**Visual distinction for auto-completed (missed) chores:**
- Shown in Completed section with strikethrough text
- Badge: "Missed" (muted color, not destructive)
- Completion count excludes auto-completed (only counts user-completed)

### Weekly Summary Report

**Access:**
1. User taps "Weekly Summary" button (visible in Weekly view)
2. Full-screen overlay appears with scrollable week grid

**Layout:**
- Column headers: Mon / Tue / Wed / Thu / Fri / Sat / Sun
- Row per recurring chore
- Cells: ✓ (completed), ✗ (missed/auto-completed), — (not scheduled)
- Color-coded by assignee using chart colors

**Design notes:**
- Weekly summary is read-only (no actions)
- Close via X button (48px) or backdrop tap
- Respects current week filter (shows This Week only, not past weeks)

## Accessibility

### Touch Target Compliance
- All interactive elements: Minimum 48×48px (exceeds WCAG 2.2 Level AAA 44px requirement)
- Chore completion checkbox: 48×48px (per CHOR-06)
- Adequate spacing between targets: Minimum 8px gap

### Keyboard Navigation
Not primary input method (kiosk is touch-only), but supported:
- Tab order follows visual order (top to bottom, left to right)
- Focus indicators: Ring visible on all focusable elements
- Enter key activates focused buttons
- Escape key closes modals
- Space toggles checkboxes

### Color Contrast
- Body text on background: 21:1 (AAA)
- Primary button text on primary: Verified contrast in preset
- Destructive button text on destructive: Verified contrast in preset
- Error text: Uses destructive color with sufficient contrast
- Family member chart colors: Verified sufficient contrast for left border accent (not primary text color)

### Semantic HTML
- Buttons use `<button>` element (not div with onClick)
- Checkboxes use `<input type="checkbox">` (via shadcn Checkbox primitive)
- Form inputs use `<input>` with associated `<label>`
- Modals use proper ARIA attributes (dialog role, aria-labelledby, aria-describedby)
- Headings use proper hierarchy (h1 → h2 → h3)
- Completed section uses `<details>` or ARIA disclosure pattern
- Edit/Delete icons include `aria-label` attributes for screen reader support

## State Management

### Component States
Each interactive element supports these states (inherited from Phase 2):

| State | Visual Treatment | When Active |
|-------|------------------|-------------|
| Default | Standard colors from design tokens | Element is idle |
| Hover | Not used (touch-only device) | N/A |
| Focus | Ring: `--ring` color, 2px offset | Keyboard navigation (secondary) |
| Active/Pressed | `scale(0.96)`, slight opacity shift | Touch down (before release) |
| Disabled | Opacity 50%, cursor not-allowed | Form validation fails, loading state |
| Loading | Spinner replaces text, disabled state | Async operation in progress |
| Error | Border: destructive color, error text below | Validation failure |
| Success | Transient toast notification | Action completes successfully |

### Chore-Specific States

| State | Visual Treatment | When Active |
|-------|------------------|-------------|
| Active Chore | Normal card, unchecked checkbox | Chore not yet completed |
| Completed Chore | Checked checkbox, opacity 0.7, in Completed section | User completed chore |
| Auto-Completed (Missed) | X checkbox, strikethrough, "Missed" badge | Recurring chore not done by deadline |
| Long-Press Active | Haptic feedback (if supported), scale(1.05) | User holding checkbox for 1000ms |
| Undo Available | Toast with "Undo" button visible | Within 5 seconds of completion |

### Form States
- Pristine: No validation, submit button enabled if required fields present
- Dirty: User has modified input, validate on blur
- Submitting: Loading state, inputs disabled, submit button shows spinner
- Success: Form clears (for Add), modal closes, toast notification
- Error: Error message shown, inputs remain editable, retry enabled

### View States
- Daily View: Filter to chores created today (DATE(created_at) = DATE('now'))
- Weekly View: Filter to chores created this week (Sunday-Saturday)
- Empty state adapts to view: "All Done for Today!" vs "No Chores This Week"

## Motion and Animation

### Transitions (Inherited from Phase 2)
- Button press: `transform 150ms ease-out`
- Modal enter: `opacity 200ms ease-out`, `scale 200ms ease-out`
- Modal exit: `opacity 150ms ease-in`, `scale 150ms ease-in`
- Toast enter: `slide-in-right 300ms ease-out`
- Toast exit: `fade-out 200ms ease-in`

### Chore-Specific Animations

**Completion Animation (D-18, user-approved):**
- Checkbox: Scale bounce `scale(0 → 1.2 → 1.0)` over 300ms ease-out
- Chore card: Fade-out `opacity(1 → 0)` over 200ms, then slide out (height collapse)

**Celebration Animation (D-27, user-approved):**
- Icon + text: Fade-in `opacity(0 → 1)` + scale `scale(0.95 → 1.0)` over 300ms
- Gentle, not distracting
- Plays once when user completes last active chore

**Long-Press Feedback:**
- Checkbox: Scale up `scale(1.0 → 1.05)` at 800ms (200ms before picker opens)
- Subtle pulse to indicate threshold approaching

**Completed Section Expand/Collapse:**
- Height: Animated with `transition-all 300ms ease-in-out`
- Rotation icon: Chevron rotates 180° when expanded

**No Animations:**
- List reordering (chores sort by points descending, no drag-and-drop)
- Hover effects (touch-only device)

**Performance:** All animations use `transform` and `opacity` (GPU-accelerated). No `left`, `top`, `width`, or `height` animations except for Completed section collapse (acceptable for infrequent action).

## Responsive Considerations

**Target device:** Raspberry Pi touchscreen (800×480 or 1024×600 typical resolutions)

**Design approach:** Fixed layout optimized for kiosk screen size. Not responsive to different viewports.

**Minimum assumptions:**
- Width: 800px minimum
- Height: 480px minimum
- Touch input only (no mouse hover)
- Landscape orientation (primary)

**Portrait mode:** Not prioritized for Phase 3. If device rotates, layout should remain functional but may not be optimized.

**Scrolling:**
- Chore list: Vertically scrollable if content exceeds viewport
- Completed section: Scrollable within collapsed container
- Add/Edit form modal: Scrollable if content exceeds viewport height
