## Spacing Scale

Declared values (Tailwind default scale, all multiples of 4):

| Token | Value | Usage |
|-------|-------|-------|
| 1 | 4px | Icon gaps, minimal inline padding |
| 2 | 8px | Compact element spacing, internal button padding |
| 4 | 16px | Default element spacing, form field gaps |
| 6 | 24px | Section padding, card internal spacing |
| 8 | 32px | Layout gaps between major sections |
| 12 | 48px | Major section breaks, page-level spacing |
| 16 | 64px | Large page-level spacing (if needed) |

**Exceptions for this phase:**
- Touch targets: Minimum 44px (11 units) for all interactive elements per FAM-07
  - Buttons: `min-h-11` (44px)
  - Form inputs: `min-h-11` (44px)
  - List item tap zones: `min-h-11` (44px)
  - Delete icons: Touch zone expanded to 44px even if icon is smaller

**Rationale:** WCAG 2.2 Level AAA requires 44×44px minimum touch targets. This is critical for kiosk touchscreen usability.

## Color

Colors use oklch color space (preset configuration). Values shown are CSS custom properties.

| Role | Light Mode Value | Usage |
|------|------------------|-------|
| **Dominant (60%)** | `--background: oklch(1 0 0)` (white) | Page background, main surface |
| **Secondary (30%)** | `--card: oklch(1 0 0)` (white), `--muted: oklch(0.97 0 0)` (light gray) | Card backgrounds, muted sections, empty states |
| **Accent (10%)** | `--primary: oklch(0.52 0.105 223.128)` (blue) | Reserved for: Primary CTA button, focus rings, active states |
| **Destructive** | `--destructive: oklch(0.577 0.245 27.325)` (red-orange) | Delete button only |

**Accent reserved for:**
1. Primary "Add Family Member" button
2. Focus ring on form inputs (keyboard navigation)
3. Active/pressed state on primary buttons
4. PIN input highlights (when entering PIN)

**NOT used for:** List items, secondary buttons, text links, icons (use muted or foreground colors instead).

**Border/Input:**
- Border: `oklch(0.922 0 0)` (light gray)
- Input border: `oklch(0.922 0 0)` (matches border)
- Focus ring: `oklch(0.708 0 0)` (medium gray, with primary accent overlay)

**Dark mode:** Supported via preset (`.dark` class) but not required for Phase 2. Kiosk runs in light mode by default.

## Interaction Patterns

### Touch Feedback
All interactive elements must provide immediate visual feedback:

```css
/* Applied to all buttons and tappable elements */
.touch-target:active {
  transform: scale(0.96);
  transition: transform 150ms ease-out;
}
```

**Requirements (FAM-08):**
- Feedback appears within 100ms of touch
- Visual change is perceivable (scale + opacity or background color shift)
- No hover states (kiosk is touch-only)

### PIN Entry Flow
1. User taps protected action (Add/Edit/Delete)
2. PIN modal appears with numeric keypad
3. User taps digits (each tap shows visual feedback)
4. PIN displays as masked dots (••••)
5. On 4th digit, auto-submit verification
6. Success: Modal closes, action proceeds
7. Failure: Error message appears, input clears, try again

**Design notes:**
- PIN pad: 3×4 grid (1-9, 0, clear, submit if needed)
- Button size: 56px×56px with 8px gap
- Modal is vertically centered, dims background
- Escape via "Cancel" button (44px) or backdrop tap

### Form Validation
- Real-time validation on blur (not on every keystroke)
- Error messages appear below field in destructive color
- Submit button disabled if validation fails
- Clear visual distinction between valid/invalid states (border color change)

## Accessibility

### Touch Target Compliance
- All interactive elements: Minimum 44×44px (WCAG 2.2 Level AAA)
- High-precision elements (PIN pad): 56×56px for better accuracy
- Adequate spacing between targets: Minimum 8px gap

### Keyboard Navigation
Not primary input method (kiosk is touch-only), but supported:
- Tab order follows visual order (top to bottom, left to right)
- Focus indicators: Ring visible on all focusable elements
- Enter key activates focused buttons
- Escape key closes modals

### Color Contrast
- Body text on background: 21:1 (AAA)
- Primary button text on primary: Verified contrast in preset
- Destructive button text on destructive: Verified contrast in preset
- Error text: Uses destructive color with sufficient contrast

### Semantic HTML
- Buttons use `<button>` element (not div with onClick)
- Form inputs use `<input>` with associated `<label>`
- Modals use proper ARIA attributes (dialog role, aria-labelledby, aria-describedby)
- Headings use proper hierarchy (h1 → h2 → h3)

## State Management

### Component States
Each interactive element supports these states:

| State | Visual Treatment | When Active |
|-------|------------------|-------------|
| Default | Standard colors from design tokens | Element is idle |
| Hover | Not used (touch-only device) | N/A |
| Focus | Ring: `--ring` color, 2px offset | Keyboard navigation (secondary) |
| Active/Pressed | `scale(0.96)`, slight opacity shift | Touch down (before release) |
| Disabled | Opacity 50%, cursor not-allowed | Form validation fails, loading state |
| Loading | Spinner replaces text, disabled state | Async operation in progress |
| Error | Border: destructive color, error text below | Validation failure |
| Success | Transient toast notification | Action completes successfully |

### Form States
- Pristine: No validation, submit button enabled if required fields present
- Dirty: User has modified input, validate on blur
- Submitting: Loading state, inputs disabled, submit button shows spinner
- Success: Form clears (for Add), modal closes, toast notification
- Error: Error message shown, inputs remain editable, retry enabled

### Data Loading States
- Initial load: Full-screen skeleton or spinner with "Loading..." text
- Empty state: Custom empty state component with heading + body + CTA
- Error state: Error message with retry button (44px)
- Loaded: Display data (family member list)

## Motion and Animation

### Transitions
- Button press: `transform 150ms ease-out`
- Modal enter: `opacity 200ms ease-out`, `scale 200ms ease-out`
- Modal exit: `opacity 150ms ease-in`, `scale 150ms ease-in`
- Toast enter: `slide-in-right 300ms ease-out`
- Toast exit: `fade-out 200ms ease-in`
- Form validation error: `shake 300ms ease-in-out` (subtle horizontal shake)

### No Animations
- Page navigation (kiosk is single-page app for this phase)
- List reordering (not a feature in Phase 2)
- Hover effects (touch-only device)

**Performance:** All animations use `transform` and `opacity` (GPU-accelerated). No `left`, `top`, `width`, or `height` animations.

## Responsive Considerations

**Target device:** Raspberry Pi touchscreen (800×480 or 1024×600 typical resolutions)

**Design approach:** Fixed layout optimized for kiosk screen size. Not responsive to different viewports.

**Minimum assumptions:**
- Width: 800px minimum
- Height: 480px minimum
- Touch input only (no mouse hover)
- Landscape orientation (primary)

**Portrait mode:** Not prioritized for Phase 2. If device rotates, layout should remain functional but may not be optimized.
