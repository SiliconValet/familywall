## Spacing Scale

Declared values (Tailwind default scale, all multiples of 4 — inherited from Phase 2):

| Token | Value | Usage |
|-------|-------|-------|
| 1 | 4px | Icon gaps, swatch-to-swatch gap in color picker row |
| 2 | 8px | Compact element spacing, gap between color swatches |
| 4 | 16px | Default element spacing, form field gaps |
| 6 | 24px | Section padding, modal internal spacing |
| 8 | 32px | Layout gaps between major sections |
| 12 | 48px | Major section breaks |
| 16 | 64px | Large page-level spacing |

**Exceptions for this phase:**
- Touch targets: Minimum 44px (Tailwind `min-h-11`) for all interactive elements — inherited from FAM-07.
- Color swatch buttons: 40px × 40px each (Tailwind `h-10 w-10`). Swatches sit in a row with 8px gap; combined row width fits within modal. Smaller than 44px but the 8px gap means adjacent tap zones do not overlap — acceptable precision tradeoff for color selection.
- Virtual keyboard keys: Minimum 44px height (`min-h-11`). Width is proportional to key label length. Done and Space keys span full width of their row.
- Keyboard overlay height: 280px fixed. Content above shifts up via `padding-bottom: 280px` applied to the scroll container while keyboard is visible.

**Source:** Spacing scale from Phase 2 UI-SPEC. Exceptions from CONTEXT.md D-05 (keyboard fixed bottom overlay) and D-08 (color swatches).

## Interaction Patterns

### PIN Session Timeout (new in Phase 6)

**Decision source:** CONTEXT.md D-16 through D-19.

- After a successful PIN verification, a 60-second grace period begins.
- During the grace period: subsequent protected actions execute immediately — no PIN modal shown.
- Timer resets on each successful authenticated action within the grace period.
- After 60 seconds of inactivity: next protected action shows PIN modal again.
- No visual countdown indicator. Session expires silently. (D-18)
- Implementation: `lastAuthTime` as `useRef<number>` in `usePinAuth`. Check `Date.now() - lastAuthTime.current < 60_000` in `withPinAuth` before deciding to show modal. (D-19)

**Rationale:** Reduces friction for parents taking multiple sequential admin actions (e.g., editing several family members back to back).

### Virtual Keyboard Behavior (new in Phase 6)

**Decision source:** CONTEXT.md D-04 through D-07, RESEARCH.md Pattern 2.

- Keyboard auto-appears when any text `<input>` receives focus — no explicit show button. (D-04)
- Keyboard renders as a fixed-bottom overlay at 280px height. (D-05)
- Content above keyboard: `padding-bottom: 280px` applied to the page scroll container while keyboard is visible, preventing content from being obscured.
- Keyboard slides up from the bottom edge with a `transform: translateY` transition, 200ms ease-out.
- Single QWERTY layout with top numbers row and backspace. (D-06)
- Dismissal: tap outside the keyboard OR press the "Done" button. (D-07)
- Keyboard excluded from: PIN pad inputs in PinModal (use `data-no-keyboard` attribute), any `type="tel"` or `type="number"` inputs.
- Keyboard renders at app root level (inside `KeyboardProvider`, outside all Dialog components) to avoid Radix focus trap capture. Source: RESEARCH.md Open Question 2.
- On input focus change: keyboard internal buffer resets via `keyboardRef.current.setInput('')`. Source: RESEARCH.md Pitfall 2.

**Layout during keyboard visible state:**
```
┌────────────────────────────────────┐
│  [Page content scrolls here]       │
│                                    │
│  (280px padding-bottom on scroll   │
│   container prevents content       │
│   being hidden behind keyboard)    │
├────────────────────────────────────┤
│  1  2  3  4  5  6  7  8  9  0  ⌫  │  ← 44px key height
│  q  w  e  r  t  y  u  i  o  p     │
│   a  s  d  f  g  h  j  k  l       │
│    z  x  c  v  b  n  m  [Space]   │
│              [    Done    ]        │  ← full-width, 48px height
└────────────────────────────────────┘
```

### Color Swatch Picker in FamilyFormModal (new in Phase 6)

**Decision source:** CONTEXT.md D-08, D-09, D-10. Color palette from RESEARCH.md Pattern 4.

- 8 circular swatch buttons displayed in a horizontal row below the Name input field.
- Each swatch: 40px × 40px (`h-10 w-10`), `border-radius: 50%`, inline `backgroundColor` from hex value.
- Selected state: 2px ring in `--primary` color, 2px offset. (`ring-2 ring-primary ring-offset-2`)
- Unselected state: no ring.
- Auto-selection on create: server assigns first unassigned palette color. Client displays that color as pre-selected.
- On edit: current member color is pre-selected.
- No null/empty color state — every member always has an assigned color.

**Focal point:** Name input field. Focus lands here on modal open. Color swatch row is below and reachable by tap; no tab-to-swatch required.

### Settings Tab Restructure (new in Phase 6)

**Decision source:** CONTEXT.md D-12 through D-15.

- Tab label: "Settings" (was "Family"). Rightmost position in navigation bar.
- Settings page layout (vertical stack, 24px between sections):
  1. **Family Members section** (existing): family member list with Add/Edit/Delete actions.
  2. **System Settings section** (new): Section heading "System Settings" at heading size (24px, semibold). Below it:
     - "Change PIN" button (secondary/outline variant, 44px height) — existing functionality, opens ChangePinModal.
     - "Exit Kiosk Mode" button (destructive variant, 44px height, full width). PIN required before executing.

**Focal point (Settings tab):** Family Members list. Page loads with focus at the top of the list. System Settings section is reached by scrolling down; no auto-focus to System Settings on page load.

**Exit Kiosk flow:**
1. User taps "Exit Kiosk Mode" button.
2. `withPinAuth` fires — if within 60s grace period, executes directly; otherwise shows PIN modal.
3. On PIN verified: dialog closes, client calls `POST /api/system/exit-kiosk`.
4. Server responds `{ success: true }`, then after 300ms delay fires `pkill -SIGTERM chromium`.
5. Chromium process terminates — kiosk exits.

**Source:** RESEARCH.md Pattern 6, CONTEXT.md D-14, D-15.

### Offline / Calendar Error Notification (new in Phase 6)

**Decision source:** CONTEXT.md D-01 through D-03.

- When `useCalendarData.fetchEvents` catches an error: fire `toast.error('Calendar unavailable — no internet connection')`.
- Toast is auto-dismissing (Sonner default). No retry button in the toast. (D-03)
- Toast type: `toast.error()` — renders with red/destructive styling.
- No persistent banner. No per-section status badge. Subsequent auto-refresh attempts are silent unless they also fail (in which case, another toast fires). (D-01)
- Chores have no offline indicator — chores use local SQLite and always work. (D-02)

### Touch Feedback (inherited from Phase 2)

All interactive elements:
- `.touch-target` class: `min-h-12 min-w-12` (48px), `-webkit-tap-highlight-color: transparent`, `user-select: none`
- Active state: `transform: scale(0.96)`, `transition: transform 150ms ease-out`
- Feedback within 100ms of touch
- No hover states (kiosk is touch-only)

Color swatch buttons: use the same active scale but scale to `0.92` (more noticeable on smaller target) to confirm selection clearly.

## Motion and Animation

**Inherited from Phase 2:**
- Button press: `transform 150ms ease-out`
- Modal enter: `opacity 200ms ease-out`, `scale 200ms ease-out`
- Modal exit: `opacity 150ms ease-in`, `scale 150ms ease-in`
- Toast enter: slide-in, 300ms ease-out
- Toast exit: fade-out, 200ms ease-in

**New in Phase 6:**
- Virtual keyboard slide-up: `transform: translateY(100%)` → `translateY(0)`, `200ms ease-out`
- Virtual keyboard slide-down (dismiss): `translateY(0)` → `translateY(100%)`, `150ms ease-in`
- Color swatch selection: `transform: scale(1.05)` pulse on tap, 100ms, snaps back immediately after ring appears

**Performance:** All animations use `transform` and `opacity` (GPU-accelerated). Keyboard animation uses `will-change: transform` while animating to prevent layout thrash.

## Accessibility

### Touch Target Compliance (inherited + updated)
- All interactive elements: Minimum 44px height (Tailwind `min-h-11`)
- Color swatches: 40px × 40px with 8px gap. Acceptable precision tradeoff (see Spacing exceptions).
- Virtual keyboard keys: Minimum 44px height
- PIN pad buttons: 56px × 56px (inherited from Phase 2)
- Exit Kiosk button: Full width, 44px height

### Keyboard Navigation (inherited)
- Tab order follows visual order
- Focus indicators visible on all focusable elements
- Enter key activates focused buttons
- Escape key closes modals and dismisses virtual keyboard

### Color Contrast
- All 8 palette colors used as event/badge backgrounds: verified accessible against white text (Tomato, Tangerine, Sage, Basil, Peacock, Blueberry) and against dark text (Banana — use `color: #000` or `--foreground` when Banana is the background color).
- Flamingo (#E67C73) is borderline — use dark text over Flamingo backgrounds to ensure WCAG AA contrast.

### Semantic HTML (inherited + updated)
- Color swatch buttons use `<button>` with `aria-label="{colorName}"` and `aria-pressed="{true/false}"` for selected state
- Delete icon button on member card: `aria-label="Delete [Name]"` where [Name] is the family member's name, populated at render time (e.g., `aria-label="Delete Alice"`)
- Settings section uses `<section>` with appropriate heading hierarchy
- Virtual keyboard keys use `<button>` elements
